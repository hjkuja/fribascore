You are setting up a lightweight multi-agent AI team system for this repository.
Your job is to: scan the existing codebase for context, remove any Squad remnants,
and scaffold a clean .council/ system based on the spec below.

## Step 1 — Scan the repository

Before creating any files, read and summarize:
- Tech stack (languages, frameworks, key dependencies)
- Project structure (main modules, layers, boundaries)
- Existing conventions (naming, patterns, error handling)
- Any existing architectural decisions you can infer from the code

Also read and extract content from any existing AI team files:
- .squad/ — read team.md, decisions.md, routing.md, and all agents/*/charter.md
  and agents/*/history.md
- .council/ — read any existing files in the same way
- .github/copilot-instructions.md — extract any project-relevant content,
  ignore Squad-specific orchestration instructions
- Any other .md files in the root that describe architecture or conventions

For each source, extract and classify content into these buckets:
- Decisions → will go into memory/decisions.md
- Agent learnings → will go into memory/history/{role}.md for the closest
  matching role (lead, backend, frontend, tester)
- Architectural knowledge → will go into memory/architecture.md
- Routing rules → will go into routing.md
- Skills or procedural knowledge → will go into skills/

Note the source of each extracted item so nothing is silently lost.
If content doesn't clearly fit any bucket, put it in memory/architecture.md
with a comment flagging it for human review.

Do not guess or invent — only carry forward what is explicitly stated in
existing files or confidently inferable from the actual code.

## File specs

### orchestrator.md
Max ~400 tokens. Covers:
- How to read a task and decide which agent(s) to activate
- Whether to go sequential or parallel (parallel when tracks are independent)
- Handoff protocol: what context to pass when switching agents
- When to escalate to lead vs go direct to specialist
- Reminder to always read memory/decisions.md and memory/architecture.md before starting

### routing.md
Explicit rules for task routing. Editable by the human. Structure:
- Per-domain ownership (e.g. "API endpoints → backend", "React components → frontend")
- Overlap rules (who leads when a task spans domains)
- Escalation triggers (when lead must be involved)
- Populate with real rules inferred from the scanned codebase

### charters/lead.md
Role: architect, decision-maker, conflict resolver, task decomposer.
Structure:
- Role summary (2 sentences)
- Owns (bullet list)
- Does NOT own (bullet list)
- Escalates to human when (bullet list)
- Decision logging: "Append decisions to memory/inbox/lead.md:
  [YYYY-MM-DD] [DECISION] what | why | alternatives rejected"
- History logging: "Append project learnings to memory/history/lead.md:
  [YYYY-MM-DD] [LEARNING] one line, actionable"
- "Load relevant skills from skills/ before starting complex tasks"

### charters/backend.md
Same structure as lead.md. Owns: API layer, data layer, integrations, auth, 
business logic. Uses memory/inbox/backend.md and memory/history/backend.md.

### charters/frontend.md
Same structure. Owns: UI components, state management, routing, UX concerns.
Uses memory/inbox/frontend.md and memory/history/frontend.md.

### charters/tester.md
Same structure. Owns: test strategy, coverage, edge cases, integration tests.
Does NOT write production code. Uses memory/inbox/tester.md and
memory/history/tester.md.

### charters/scribe.md
Invoked manually by the human at end of session. Instructions:
1. Read all memory/inbox/*.md files
2. Read current memory/decisions.md
3. Deduplicate — skip any decision already captured in decisions.md
4. Append new decisions to memory/decisions.md in chronological order
5. Clear each inbox file (leave file, empty the content)
6. Output a short summary: how many decisions merged, from which agents,
   anything that looked contradictory or worth human review

### memory/decisions.md
Pre-populate with any real decisions you can confidently infer from the codebase.
Format per entry:
---
[YYYY-MM-DD] [DECISION]
What: ...
Why: ...
Alternatives rejected: ...
Source: inferred from codebase scan
---

If nothing can be confidently inferred, leave a clear empty template with just
the format comment.

### memory/architecture.md
Pre-populate from scan. Sections:
- Stack (languages, frameworks, runtimes)
- Structure (module layout, layer boundaries)
- Key patterns (conventions observed in code)
- Integration points (external services, APIs, databases)
- Known constraints or non-obvious rules

### memory/inbox/*.md
All start empty. Add a one-line comment at top:
`<!-- Agents append decisions here during sessions. Cleared by scribe. -->`

### memory/history/*.md
All start empty. Add a one-line comment at top:
`<!-- {Role} appends project-specific learnings here. Never cleared. -->`

### skills/.gitkeep
Empty placeholder. Skills are added by the human or agents over time as
procedural playbooks for recurring tasks in this project.

### USAGE.md
Write a practical guide covering:
- How to start a session (which file to give as context, how to activate an agent)
- How to talk to a specific agent directly vs using the orchestrator
- How to hand off between agents mid-task
- How and when to invoke the scribe
- How to add a new skill
- When to update routing.md manually
- A concrete end-to-end example using the actual tech stack found in this repo

## Constraints

- Every charter must be self-contained: injectable as a system prompt prefix
- Do not invent facts about the codebase — only use what you actually found
- Keep orchestrator.md and each charter concise — they will be loaded into context
  on every session, so token cost matters
- All files must be plain markdown, no custom syntax
- The system must work with Claude Code, GitHub Copilot agent mode, or any
  other AI coding agent — no tool-specific references in the files themselves
